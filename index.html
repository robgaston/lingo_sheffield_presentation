Semantic Modeling with Arches

We model vocabulary data as Arches resources: ontology-backed, polyhierarchical, self-describing.

  • Concepts and schemes are Arches resource models built on the LinkedArt+ ontology
  • Polyhierarchy, labels, notes, attribution; all modeled semantically in tiles
  • What you get for free: edit log, lifecycle states, ETL, permissions
  • The trade-off: you have to work with Arches data structures (tiles, nodegroups, JSONB)

Takeaway: semantic modeling gives you a complete, self-describing data layer; you need good tooling to work with it efficiently.

- This is the fundamental architectural decision: vocabulary data lives in Arches resource models - The LinkedArt+ ontology gives us a complete, well-defined structure - Emphasize the trade-off: you get everything for free, but you pay in complexity of data access - This trade-off is what motivated arches-querysets (next slide) - ~60s

Tile APIs with arches-querysets

arches-querysets eliminated one of our biggest pain points: working directly with Arches tiles.

# Before: raw tile queries by UUID
tile = TileModel.objects.get(
    resourceinstance_id=concept_id,
    nodegroup_id="9c2a237c-..."
)
label = tile.data["4fa811a5-..."][0]["value"]

# After: aliased_data via arches-querysets
tree = ResourceTileTree.get_tiles("concept", resource_id=concept_id)
label = tree.aliased_data.appellative_status[0] \
    .aliased_data.appellative_status_ascribed_name_content
- Let the code speak; just highlight the before/after contrast - "If you've ever written UUID-based tile queries, you know the pain" - The aliased_data API gives you named traversal instead of UUID soup - Lingo funded much of the arches-querysets development, so the two projects evolved together - Transition: "And it integrates beautifully with DRF"

arches-querysets: DRF Integration

  • LingoResourceSerializer / LingoTileSerializer extend arches-querysets base classes
  • All tile CRUD → DRF generic views (ArchesTileListCreateView, ArchesTileDetailView)
  • Custom validation hooks (e.g., SKOS prefLabel uniqueness) live in the serializer

Takeaway: if you're writing tile-level APIs, start with arches-querysets. The boilerplate reduction is dramatic.

- Key message: arches-querysets isn't just a convenience: it's the right abstraction layer for tile APIs - Mention: custom validation hooks let you enforce SKOS rules (like prefLabel uniqueness per language) in the serializer - Transition: "So we had good APIs. What did we build on top of them?"

A Full SPA on Arches

Lingo replaces Arches' resource-centric report/editor views with a standalone Vue Router SPA.

  • Vue Router 4 handles all client-side navigation
  • Arches serves a single HTML shell (LingoRootView) for all routes
  • You still get auth, notifications, and other APIs from core Arches
  • Pages are domain-specific: hierarchy browser, concept editor, search, dashboard

Takeaway: you don't have to use Arches' default report templates. A custom SPA gives you full control while still sitting inside the Arches shell.

- Point: Arches gives you a Django view that serves an HTML shell. From there, Vue Router takes over entirely. - You still get auth, nav, notifications; the Arches chrome is there, but your app owns the content area - This is genuinely a full SPA experience for the user - Transition: "Let me show you what this looks like"

Search Without Elasticsearch

Lingo implements its own search backend entirely in PostgreSQL.

Quick search:

-- pg_trgm GIN index on label tile content
SELECT resourceinstanceid,
       similarity(label_value, 'censer') AS sim
FROM label_tiles
WHERE label_value % 'censer'
   OR label_value ILIKE '%censer%'
ORDER BY rank_tier, sim DESC
  • 8-tier relevance ranking: label type × match position
  • Language priority as tiebreaker
  • Wrapped in SearchResultSet - compatible with Django's Paginator
- Key point: we don't use Elasticsearch at all. Pure PostgreSQL with pg_trgm. - The GIN index on label tiles gives us fast fuzzy matching - 8-tier ranking means prefLabels and exact matches float to the top - "This handles 500K+ terms in the Getty AAT without breaking a sweat" - Transition: "For more complex queries, we built a composable search tree"

Advanced Search: Query Tree

AdvancedSearchEvaluator: JSON query tree → composable Django QuerySet chains

  • AND = nested filter(pk__in=...)
  • OR = Q(pk__in=...) | Q(pk__in=...)

Hierarchy cascade - single recursive CTE:

WITH RECURSIVE descendants AS (
    SELECT resourceinstanceid FROM edge_list
    WHERE broader_id = %s
    UNION ALL
    SELECT e.resourceinstanceid FROM edge_list e
    JOIN descendants d ON e.broader_id = d.resourceinstanceid
)
SELECT resourceinstanceid FROM descendants;
- The recursive CTE is the key insight: hierarchy cascading as a SQL query - AdvancedSearchEvaluator composes these into arbitrary boolean trees - "You can say: find all concepts under X that have a German label containing 'Kirche'" - ~90s on this one; the code is dense, take time to explain

Frontend State Management

Two patterns that prevented an entire category of bugs:

useResourceStore: surgical refresh:

// After saving a nodegroup, refresh only that data
await saveNodegroup("appellative_status", tileId, payload);
await resourceStore.refreshNodegroup("appellative_status");
// Falls back to full resource refresh on error
  • Prevents redundant API calls from sibling edit components
  • Single reactive cache per page for all resource data
- "Two patterns that sound simple but prevented entire categories of bugs" - Surgical refresh: after saving a nodegroup, only re-fetch that slice, not the whole resource - At AAT scale, re-fetching everything would be brutal (58K concepts) - Transition: "The other pattern is about duplicate API calls"

Frontend: Inflight Deduplication

useConceptStore: shared fetches:

// Two components request the same children → one fetch
if (inflightChildFetches.has(conceptId)) {
    return inflightChildFetches.get(conceptId);
}
const promise = fetchChildren(conceptId);
inflightChildFetches.set(conceptId, promise);

Takeaway: in a Vue 3 app with complex resource data, design your state management strategy early. These patterns are reusable.

- A Map of in-flight promises: if the same fetch is already running, just return that promise - "Expanding a tree node can trigger the same API call from multiple components simultaneously" - Simple pattern, but eliminated a whole class of duplicate-request bugs - Transition: "OK, shifting gears, let's talk about permissions"

Simplified Permissions

Lingo opts out of Arches' full node-level RBAC:

class ReadOnlyOrLingoEditor(BasePermission):
    def has_permission(self, request, view):
        if request.method in SAFE_METHODS:
            return True
        return request.user.groups.filter(
            name="Lingo Editor"
        ).exists()
  • One Django group: "Lingo Editor"
  • Non-editors get read-only; anonymous access configurable
  • Appropriate when the model is "editors vs. readers"

Takeaway: core Arches nodegroup-level permissions require costly graph lookups per request. If your access model is simpler, a lightweight DRF permission class skips all of that.

- This is a good example of the "bypass generic features" philosophy - Core Arches nodegroup permissions hit the graph on every request. That's expensive. - Lingo's model is dead simple: you're an editor or you're not. One group check. - "Don't assume you need the full RBAC machinery. Think about what your domain actually requires." - ~60s

Identifiers & Linked Open Data

Lingo generates stable URIs for concepts when a scheme is published.

  • Identifier rules are defined per scheme: pattern, namespace, authority
  • On the "publish" lifecycle event, URIs get minted for every concept in the scheme
  • Built on Arches 8.1's custom typed resource identifiers: typed IDs you can define for any application need
  • Result: your vocabularies are shareable as linked open data with dereferenceable URIs

Takeaway: lifecycle events + typed identifiers let Lingo produce publication-ready linked open data with no external tooling.

- Walk through the flow: you define identifier rules on the scheme, hit publish, URIs get minted - This uses Arches 8.1's custom typed resource identifiers: a core feature we can plug into - The URIs are dereferenceable, so your vocabulary can participate in the linked data ecosystem - "This was one of the headline requirements for Lingo 1.0: proper LOD publishing" - ~60s

UX Lessons from Real Editorial Use

Using the tool for real vocabulary management surfaced improvements no test plan would have caught:

  • Concept sets is a workflow tool, not just bookmarks
  • Concepts displayed throughout the UI need to be rendered as links to allow for quick navigation
  • Advanced search needed attribution facets (find by source/contributor)

Takeaway: Get real users doing real work as early as possible. Test scenarios model what you expect; real use reveals what you missed.

- "We had editors using Lingo for real vocabulary work months before 1.0" - Each bullet is something we wouldn't have built without that feedback - Concept sets: editors needed to group concepts for batch operations, not just bookmark them - Lifecycle retirement: edge cases only emerge after months of sustained use - ~60s - Transition: "Let me distill all of this into some concrete lessons"

Lessons: Architecture

  • If your domain fits Arches resource models, use them; you get so much infrastructure for free
  • Design your graphs carefully though; they're load-bearing and hard to change later
  • arches-querysets should be your starting point for tile-level APIs
- "These are the architecture lessons I wish someone had told me two years ago" - Graph design: emphasize that changing graphs after you have data is painful. Get it right early. - ~45s - Transition: "What about working with extensions?"

Lessons: Extensions & Ecosystem

  • Compose extensions deliberately; your app can consume several at once
  • When timelines don't align, build behind clean abstractions and plan for convergence
  • Stable APIs and semantic versioning matter enormously; breaking changes cascade into your release timeline
  • Extension developers: think about downstream consumers. Hooks and extension points are hard to align across projects
- "This is both an app developer lesson and an extension developer lesson" - The arches-search story is the concrete example: we shared direction but not timeline - Stable APIs: if you're building an extension others will use, semver is non-negotiable - ~45s - Transition: "And on performance..."

Lessons: Performance

  • Test with ambitious data early; an order of magnitude breaks assumptions
  • Defer everything possible: load only what's visible, lazy-load the rest
  • Custom apps can skip generic core features (reporting, nodegroup permissions) that require many graph-lookup queries
  • PostgreSQL (pg_trgm, GIN indexes, recursive CTEs) is powerful, but be mindful that GIN indexes on tiles compound write overhead across apps
- "Test with ambitious data": can't say this enough. Our assumptions broke at 5x scale. - The generic features point: core Arches does a lot of graph resolution per request. If your app knows its structure, you can skip that. - GIN index concern: be honest that this is a real risk if the ecosystem grows - ~60s
