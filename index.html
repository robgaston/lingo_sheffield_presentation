Semantic Modeling with Arches

Arches' ontology-backed resource models let us describe polyhierarchical vocabulary data in a complete, self-describing way.

  • Concepts and schemes are designed as Arches resource models based on the LinkedArt+ ontology
  • Polyhierarchy, labels, notes, attribution; modeled semantically in tiles
  • Benefit: edit log, lifecycle states, ETL, permissions come for free
  • Trade-off: we must work with Arches data structures (tiles, nodegroups, JSONB)

Takeaway: semantic modeling gives you a complete, self-describing data layer, but you need good tooling to work with it efficiently.

Tile APIs with arches-querysets

arches-querysets eliminated the biggest pain point of building on Arches tiles.

# Before: raw tile queries by UUID
tile = TileModel.objects.get(
    resourceinstance_id=concept_id,
    nodegroup_id="9c2a237c-..."
)
label = tile.data["4fa811a5-..."][0]["value"]

# After: aliased_data via arches-querysets
tree = ResourceTileTree.get_tiles("concept", resource_id=concept_id)
label = tree.aliased_data.appellative_status[0] \
    .aliased_data.appellative_status_ascribed_name_content

arches-querysets: DRF Integration

  • LingoResourceSerializer / LingoTileSerializer extend arches-querysets base classes
  • All tile CRUD → DRF generic views (ArchesTileListCreateView, ArchesTileDetailView)
  • Custom validation hooks (e.g., SKOS prefLabel uniqueness) live in the serializer

Takeaway: if you're writing tile-level APIs, start with arches-querysets. The boilerplate reduction is dramatic.

A Full SPA on Arches

Lingo replaces Arches' resource-centric report/editor views with a standalone Vue Router SPA.

  • Vue Router 4 handles all client-side navigation
  • Arches serves a single HTML shell (LingoRootView) for all routes
  • Arches shell still provides auth, notifications, nav chrome
  • Pages are domain-specific: hierarchy browser, concept editor, search, dashboard

Takeaway: you don't have to use Arches' default report templates. A custom SPA with Vue Router gives you full control while still sitting inside the Arches shell.

Search Without Elasticsearch

Lingo implements its own search backend entirely in PostgreSQL.

Quick search:

-- pg_trgm GIN index on label tile content
SELECT resourceinstanceid,
       similarity(label_value, 'architectu') AS sim
FROM label_tiles
WHERE label_value % 'architectu'
   OR label_value ILIKE '%architectu%'
ORDER BY rank_tier, sim DESC
  • 8-tier relevance ranking: label type × match position
  • Language priority as tiebreaker
  • Wrapped in SearchResultSet - compatible with Django's Paginator

Advanced Search: Query Tree

AdvancedSearchEvaluator: JSON query tree → composable Django QuerySet chains

  • AND = nested filter(pk__in=...)
  • OR = Q(pk__in=...) | Q(pk__in=...)

Hierarchy cascade - single recursive CTE:

WITH RECURSIVE descendants AS (
    SELECT resourceinstanceid FROM edge_list
    WHERE broader_id = %s
    UNION ALL
    SELECT e.resourceinstanceid FROM edge_list e
    JOIN descendants d ON e.broader_id = d.resourceinstanceid
)
SELECT resourceinstanceid FROM descendants;

The arches-search Decision

future

  • arches-search: same PostgreSQL-over-ES direction, driven by arches-rascolls
  • Still pre-release; won't initially support Lingo's domain needs (recursive hierarchy)
  • Decision: build our own behind clean abstractions, accept the debt, converge later
  • Concern: Lingo adds GIN indexes on tiles; if many apps do this, write overhead compounds
  • arches-search externalizes indexed data, a better long-term pattern for the ecosystem

Takeaway: composability is a goal, not a guarantee. When an extension shares your direction but not your timeline, build behind clean interfaces so you can swap later.

Frontend State Management

Two patterns that prevented an entire category of bugs:

useResourceStore: surgical refresh:

// After saving a nodegroup, refresh only that data
await saveNodegroup("appellative_status", tileId, payload);
await resourceStore.refreshNodegroup("appellative_status");
// Falls back to full resource refresh on error
  • Prevents redundant API calls from sibling edit components
  • Single reactive cache per page for all resource data

Frontend: Inflight Deduplication

useConceptStore: shared fetches:

// Two components request the same children → one fetch
if (inflightChildFetches.has(conceptId)) {
    return inflightChildFetches.get(conceptId);
}
const promise = fetchChildren(conceptId);
inflightChildFetches.set(conceptId, promise);

Takeaway: in a Vue 3 app with complex resource data, design your state management strategy early. These patterns are reusable.

Simplified Permissions

Lingo opts out of Arches' full node-level RBAC:

class ReadOnlyOrLingoEditor(BasePermission):
    def has_permission(self, request, view):
        if request.method in SAFE_METHODS:
            return True
        return request.user.groups.filter(
            name="Lingo Editor"
        ).exists()
  • One Django group: "Lingo Editor"
  • Non-editors get read-only; anonymous access configurable
  • Appropriate when the model is "editors vs. readers"

Takeaway: core Arches nodegroup-level permissions require costly graph lookups per request. If your access model is simpler, a lightweight DRF permission class avoids that overhead entirely.

Identifiers & Linked Open Data

Lingo generates stable URIs for concepts when a scheme is published.

  • Identifier rules are defined per scheme - pattern, namespace, authority, sequence
  • On "publish" lifecycle event, URIs are minted for each concept in the scheme
  • Built on Arches 8.1's custom typed resource identifiers - typed IDs for any application need
  • Enables sharing vocabularies as linked open data with dereferenceable URIs

Takeaway: lifecycle events + typed identifiers let Lingo produce publication-ready LOD without external tooling.

UX Lessons from Real Editorial Use

Using the tool for real vocabulary management surfaced improvements no test plan would have caught:

  • Concept sets → a workflow tool, not just bookmarks
  • Hierarchical position → clickable navigation, not just display
  • Advanced search → attribution facets (find by source/contributor)
  • Lifecycle retirement → edge cases only visible in sustained use

Takeaway: Get real users doing real work as early as possible. Test scenarios model what you expect; real use reveals what you missed.

Lessons: Architecture

  • If your domain fits Arches resource models, use them! (infrastructure for free)
  • Design your graphs carefully; they're load-bearing and hard to change
  • arches-querysets should be your starting point for tile-level APIs

Lessons: Extensions & Composition

  • Compose extensions deliberately; an app can consume multiple extensions
  • When timelines don't align, build behind clean abstractions
  • Be honest about maintenance cost vs. waiting; plan for convergence

Lessons: Performance

  • Test with ambitious data early; an order of magnitude breaks assumptions
  • Defer everything: load only what's visible, lazy-load the rest
  • Custom apps can bypass generic core features (reporting, nodegroup-level permissions) that require graph-lookup queries; purpose-built code paths can be faster
  • PostgreSQL (pg_trgm, recursive CTEs, GIN indexes) is a powerful alternative to ES
  • But: GIN indexes on the tiles table compound write overhead; if many apps add them, tile operations slow for everyone

Lessons for Extension Developers

From the perspective of an application that consumes your extensions:

  • Stable public APIs and semantic versioning matter enormously
  • Breaking changes cascade into application release timelines
  • Extension points and hooks matter, but can be hard to align across projects; Lingo couldn't adopt arches-search in 1.0.0 due to this problem
  • Think about your downstream consumers when designing APIs

The Future

  • 1.1.0: AAT-scale performance + editorial UX improvements
  • arches-search convergence: when extension points support Lingo's domain facets
  • Externalized indexes: decouple search indexes from the tiles table to avoid compounding GIN overhead across apps

  • Your applications: Lingo's patterns are reusable
